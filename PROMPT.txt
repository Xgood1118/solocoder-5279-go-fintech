需求：Go 金融领域 API（纯后端），账户 / 转账 / 风控 / 审计 / 合规 / 报表。

背景：组里调研金融行业 API 设计时发现，国内银行 / 支付 / 消金都有相似的业务模式（账户、转账、流水、对账、风控、审计）。我们想实现一个教学级金融后端，把这些通用业务跑通。技术栈 Go（chi router）+ SQLite（不上生产 DB），单二进制部署，REST + gRPC 双协议。

核心功能：
- 账户：开户（个人 / 企业）、销户、冻结 / 解冻、查询余额、交易密码
- 转账：行内转账（实时）、跨行转账（异步清算，mock 央行接口）、附言
- 流水：每笔交易生成流水（账户 / 金额 / 方向 / 类型 / 余额快照 / 时间）
- 余额：维护"账户余额表"（冗余字段）和"流水表"，对账时两者必须一致
- 风控：黑名单、限额（单笔 / 日累计 / 月累计）、可疑交易识别（短时间内大额 / 频繁转账）
- 审计：所有写操作（开户 / 转账 / 改密 / 绑卡）记录审计日志
- 合规：大额上报（单笔 > 5 万）、可疑上报（5 条 / 日）
- 报表：账户余额表、日 / 周 / 月流水表、对账差异表
- 利率：活期按日计息，月末结息；定期到期自动续存或转活期
- 协议：REST API + gRPC（同一业务双协议）

涉及模块：
- `internal/account/`：账户服务、余额计算、冻结逻辑
- `internal/transfer/`：转账服务、幂等、跨行清算 mock
- `internal/ledger/`：流水服务、双写一致性
- `internal/risk/`：风控引擎、限额、黑名单
- `internal/audit/`：审计日志、AOP 拦截
- `internal/compliance/`：合规上报
- `internal/report/`：报表聚合
- `internal/grpc/`：gRPC 服务定义
- `pkg/money/`：金额 Decimal 工具

注意事项：
- 金额用 `shopspring/decimal`（不用 float64），所有运算走 Decimal
- 余额表 + 流水表要"双写"：要么都成功要么都失败，事务保证
- 转账幂等：客户端带 `biz_id`，服务端 `INSERT ... ON CONFLICT DO NOTHING` 防重复
- 余额计算：不允许 UPDATE 后结果为负，`WHERE balance + ? >= 0` 条件更新
- 跨行转账走"清算接口"（mock）：先冻结 → 发清算 → 收回执 → 确认；任一环节失败回滚
- 风控要在事务前还是事务后？事务前：先校验限额 → 事务内扣减；不能事务内扣完再校验
- 黑名单用 Redis Set（O(1) 查询），启动时加载到内存
- 限额计数器用 Redis INCR + EXPIRE，实时校验
- 审计日志不可删：AOP 拦截写操作，单独 goroutine 写（不阻塞主事务）
- 大额 / 可疑上报用 mock，模拟央行接口；上报失败重试 3 次
- 日 / 月报表用物化视图 / 预聚合表，每天 0 点 cron 跑（不要每次实时 SUM）
- gRPC 用 protobuf 严格定义字段类型（int64 / string / decimal）
- REST 返回 RFC 7807 Problem Details 格式
- 跨日切账（每月 1 号 0 点）加分布式锁（Redis SETNX）防重复跑
- 利率按日计：日利率 = 年利率 / 360，月末结息 + 入账流水
- 测试用 testify，覆盖关键路径（转账 / 限额 / 风控）
- 性能压测：1000 TPS 持续 10 分钟不丢请求
